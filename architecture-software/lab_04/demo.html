Контрольні питання — повні відповіді

Відповіді прив'язані до коду проекту.
1. Що таке API? Що таке WebAPI?
API (Application Programming Interface) — це контракт між двома програмами: набір функцій, методів або ендпоінтів, через які одна програма може взаємодіяти з іншою, не знаючи її внутрішньої реалізації.

Web API — це API, доступне через мережу за протоколом HTTP. Клієнт надсилає HTTP-запит, сервер повертає відповідь у форматі JSON або XML.

У проекті: src/pl/server.ts реєструє маршрути /api/content і /api/storages — це і є Web API. Будь-який клієнт (браузер, Postman, мобільний додаток) може звертатись до нього.
2. Які задачі вирішує та в чому особливості рівня представлення PL?
Задачі PL:
— Приймати HTTP-запити і повертати HTTP-відповіді;
— Десеріалізувати вхідні дані (JSON → об'єкт) і серіалізувати вихідні (об'єкт → JSON);
— Маршрутизувати запити до потрібного контролера;
— Перекладати винятки BLL у відповідні HTTP статус-коди (400, 404, 500);
— Маппити Request-моделі у DTO для BLL і назад.

Особливість: PL не містить бізнес-логіки — він лише транспортний шар. У проекті ContentController тільки викликає сервіс і маппить результат, жодних перевірок у ньому немає.
3. Які типові шаблони проектування існують для реалізації рівня представлення PL?
— MVC (Model-View-Controller) — Controller обробляє запит, Model містить дані, View рендерить HTML. Класичний серверний рендеринг.
— MVP (Model-View-Presenter) — View пасивна, Presenter містить логіку представлення, тестується без UI.
— MVVM (Model-View-ViewModel) — ViewModel має двостороннє binding з View. Використовується у WPF, Angular, Vue.
— REST Controller (без View) — те що в цьому проекті. Контролер повертає JSON замість HTML, клієнт сам рендерить.

У проекті реалізовано патерн REST Controller — ContentController і StorageController без жодного View.
4. В чому особливість та як реалізується шаблон MVC?
MVC розділяє відповідальності на три компоненти:

— Model — дані і бізнес-логіка (відповідає BLL + DAL у цьому проекті);
— View — шаблон для рендерингу HTML (у Web API замінюється JSON-серіалізацією);
— Controller — приймає запит, викликає Model, передає результат у View.

Особливість: View і Controller залежать від Model, але Model не знає про View — це однонаправлена залежність.

Реалізація: В ASP.NET MVC Controller успадковує Controller, методи позначені атрибутами [HttpGet], [HttpPost]. В Express аналогічно — router.get(), router.post() у ContentController.ts.
5. В чому особливість та як реалізується шаблон MVP?
MVP — розвиток MVC для UI-додатків (WinForms, Android):

— Model — дані і бізнес-логіка;
— View — пасивна, тільки відображає і делегує дії Presenter через інтерфейс (IView);
— Presenter — містить логіку представлення, підписується на події View, оновлює її через інтерфейс.

Ключова відмінність від MVC: View нічого не знає про Model — Presenter є посередником. View тестується через mock-інтерфейс.

Реалізація: View реалізує інтерфейс IView, Presenter отримує його через конструктор. При натисканні кнопки View викликає presenter.onButtonClick(), а не сама обробляє логіку.
6. В чому особливість та як реалізується шаблон MVVM?
MVVM — патерн для UI з двостороннім зв'язуванням даних (Angular, Vue, WPF):

— Model — бізнес-дані і логіка;
— View — HTML-шаблон з binding-виразами ({{ title }}, [(ngModel)]);
— ViewModel — містить стан і команди, реалізує INotifyPropertyChanged. View автоматично оновлюється при зміні ViewModel.

Ключова особливість: двостороннє binding — зміна у View оновлює ViewModel і навпаки. Presenter в MVP оновлює View вручну, у MVVM це відбувається автоматично через механізм binding.
7. Опишіть весь шлях обробки запиту від браузера до рівня представлення (включно) з MVC.
1. Браузер відправляє HTTP-запит: GET /api/content/1
2. TCP/IP доставляє пакет до сервера на порт 3000
3. Express (веб-сервер) приймає запит
4. Middleware pipeline: cors() → express.json() (парсинг тіла)
5. Маршрутизатор порівнює URL з зареєстрованими маршрутами → знаходить GET /api/content/:id
6. Викликається обробник у ContentController
7. Контролер витягує id з req.params, викликає service.getById(id)
8. BLL повертає DTO → PL-маппер конвертує у Response-модель
9. res.json(response) → Express серіалізує у JSON і відправляє HTTP-відповідь 200
10. Браузер отримує JSON
8. Чим відрізняється ASP.NET MVC та ASP.NET WebAPI?
ASP.NET MVC — повертає HTML-сторінки через View (Razor). Призначений для веб-додатків з серверним рендерингом. Контролер успадковує Controller, метод повертає IActionResult → View.

ASP.NET Web API — повертає JSON/XML. Призначений для API, якими користуються клієнтські додатки (SPA, мобільні). Контролер успадковує ControllerBase (без View). Атрибут [ApiController] додає автоматичну валідацію моделі.

В .NET 6+ обидва об'єднані в один фреймворк. Аналог у проекті: Express з res.json() — це Web API стиль.
9. Чим відрізняється WebAPI (як рівень представлення) від усіх інших клієнт-серверних застосувань?
Традиційний веб-додаток — сервер сам рендерить HTML і повертає готову сторінку. Клієнт — тільки браузер.

Web API — сервер повертає тільки дані (JSON). Рендеринг — відповідальність клієнта. Клієнтом може бути браузер (React/Angular), мобільний додаток, інший сервіс, Postman.

Ключові відмінності:
— Немає стану сесії (stateless) — кожен запит самодостатній;
— Немає прив'язки до конкретного клієнта — один API обслуговує усіх;
— Формат відповіді — JSON, а не HTML;
— Версіонування через URL (/api/v1/) або заголовки.
10. Які види серверів ви знаєте?
— Веб-сервер — обробляє HTTP-запити, роздає статичні файли (Apache, Nginx);
— Сервер застосувань — виконує бізнес-логіку, динамічно генерує відповіді (Node.js + Express, Tomcat, IIS);
— База даних — зберігає і повертає дані (PostgreSQL, MySQL, SQLite);
— Файловий сервер — зберігає файли (FTP, S3);
— Поштовий сервер — обробляє email (SMTP, IMAP);
— Проксі-сервер — посередник між клієнтом і сервером (Nginx як reverse proxy);
— DNS-сервер — перетворює домени в IP-адреси.
11. Що таке веб-сервер і які його основні задачі? Наведіть приклади.
Веб-сервер — програмне забезпечення, яке приймає HTTP/HTTPS запити і повертає відповіді.

Основні задачі:
— Обробка HTTP-запитів і формування відповідей;
— Роздача статичних файлів (HTML, CSS, JS, зображення);
— SSL/TLS термінація (HTTPS);
— Логування запитів;
— Балансування навантаження (Nginx).

Приклади: Apache HTTP Server, Nginx, Microsoft IIS.
У проекті роль веб-сервера виконує Express.js — він слухає порт 3000 і приймає запити.
12. Що таке сервер застосувань і які його основні задачі? Наведіть приклади.
Сервер застосувань — середовище виконання, яке виконує бізнес-логіку додатку, обробляє транзакції, підключається до БД.

Основні задачі:
— Виконання коду застосунку (наш TypeScript/Node.js код);
— Управління з'єднаннями з БД (connection pool);
— Обробка сесій і автентифікація;
— Управління транзакціями.

Приклади: Node.js, Apache Tomcat (Java), WildFly, IIS (ASP.NET).
У проекті: Node.js — сервер застосувань, Express — веб-фреймворк поверх нього. Часто ці ролі суміщені в одному процесі.
13. Поясніть протокол HTTP та його роботу.
HTTP (HyperText Transfer Protocol) — протокол прикладного рівня для передачі даних між клієнтом і сервером поверх TCP/IP.

Як працює:
1. Клієнт встановлює TCP-з'єднання з сервером (порт 80 або 443 для HTTPS);
2. Відправляє HTTP-запит: рядок запиту (GET /api/content HTTP/1.1), заголовки (Content-Type: application/json), тіло (для POST/PUT);
3. Сервер обробляє і повертає відповідь: статус-рядок (HTTP/1.1 200 OK), заголовки, тіло (JSON);
4. З'єднання закривається (HTTP/1.0) або залишається (keep-alive у HTTP/1.1).

Особливість: HTTP — stateless, кожен запит незалежний. Стан зберігається через cookies або токени.
14. В чому різниця між основними HTTP методами: GET, POST, PUT, DELETE?
— GET — отримати ресурс. Без тіла запиту. Ідемпотентний (повторний виклик = той самий результат). GET /api/content/1

— POST — створити новий ресурс. Тіло містить дані. Не ідемпотентний (повторний виклик створює ще один ресурс). Повертає 201. POST /api/content

— PUT — оновити існуючий ресурс повністю. Ідемпотентний. PUT /api/content/10

— PATCH — частково оновити ресурс (одне поле). PATCH /api/content/7/availability

— DELETE — видалити ресурс. Ідемпотентний. Повертає 204. DELETE /api/content/10
15. Поясніть основні групи статус-кодів HTTP.
— 1xx — Інформаційні: запит прийнято, обробка продовжується (рідко використовуються).

— 2xx — Успіх: 200 OK (запит виконано), 201 Created (ресурс створено), 204 No Content (успіх без тіла — наш DELETE).

— 3xx — Перенаправлення: 301 Moved Permanently, 302 Found.

— 4xx — Помилка клієнта: 400 Bad Request (невірні дані — наша валідація BLL), 404 Not Found (ресурс не знайдено), 401 Unauthorized, 403 Forbidden.

— 5xx — Помилка сервера: 500 Internal Server Error (необроблений виняток).
16. В чому полягає особливість реалізації DI для ASP.NET MVC чи WebAPI застосувань?
В ASP.NET вбудований IoC-контейнер: сервіси реєструються у Program.cs через builder.Services.AddScoped<IService, Service>(), а фреймворк сам ін'єктує їх у конструктор контролера при кожному запиті.

У цьому проекті (Express) вбудованого DI немає, тому реалізований вручну через container.ts: функція createContainer() створює UnitOfWork, передає у сервіси, а сервіси передаються у контролери при реєстрації роутерів у server.ts.

Перевага DI: контролер не знає як створюється сервіс, залежить тільки від інтерфейсу — легко замінити реалізацію або підставити mock у тестах.
17. Поясніть особливості використання Postman, Fiddler та подібних інструментів.
Postman — GUI-клієнт для тестування HTTP API:
— Відправляє будь-які HTTP-запити (GET, POST, PUT, DELETE, PATCH);
— Дозволяє задавати заголовки, тіло (JSON, form-data), query-параметри;
— Зберігає запити у колекції для повторного використання;
— Показує відповідь: статус-код, заголовки, тіло, час відповіді.

Fiddler — HTTP-проксі для перехоплення трафіку:
— Показує всі HTTP-запити що йдуть з браузера/додатку;
— Дозволяє модифікувати запити і відповіді "на льоту";
— Корисний для дебагу та аналізу мережевого трафіку.

Різниця: Postman — для ручного тестування API. Fiddler — для перехоплення і аналізу трафіку.
18. В чому різниця між веб-застосуванням та веб-сервісом?
Веб-застосування — призначене для людей. Повертає HTML-сторінки, має UI, взаємодія через браузер. Приклад: інтернет-магазин, соцмережа.

Веб-сервіс — призначений для програм. Повертає структуровані дані (JSON, XML), немає UI, взаємодія machine-to-machine. Приклад: наш /api/content.

Практично: один і той самий бекенд може бути і веб-застосуванням (якщо є сторінки), і веб-сервісом (якщо є API). Сучасна архітектура зазвичай: окремий фронтенд (SPA) + окремий веб-сервіс (REST API).
19. Що таке REST?
REST (Representational State Transfer) — архітектурний стиль для побудови розподілених систем, сформульований Roy Fielding у 2000 році.

Основні принципи:
— Stateless — сервер не зберігає стан між запитами;
— Client-Server — чітке розділення клієнта і сервера;
— Uniform Interface — єдиний інтерфейс: ресурси ідентифікуються URL, дії — HTTP-методами;
— Cacheable — відповіді можна кешувати;
— Layered System — клієнт не знає скільки шарів між ним і даними.

У проекті: /api/content — ресурс, GET/POST/PUT/DELETE — дії над ним. Це REST.
20. В чому відмінність REST від RPC та SOAP?
REST — ресурсо-орієнтований. URL описує ресурс (/content/1), HTTP-метод — дію. Легкий, JSON, без схеми.

RPC (Remote Procedure Call) — дієво-орієнтований. URL описує дію (/getContentById, /deleteContent). Завжди POST. Приклад: gRPC, JSON-RPC.

SOAP — протокол на XML. Суворо типізований через WSDL-схему. Підтримує транзакції і безпеку (WS-Security). Важкий, але надійний для enterprise.

Порівняння: REST — простий і гнучкий, добре для публічних API. SOAP — складний, але з гарантіями, добре для банків і корпоративних систем. RPC — коли важливіша дія, ніж ресурс (gRPC — швидкий для мікросервісів).
21. Як побудована маршрутизація в ASP.NET WebAPI та як вона працює?
В ASP.NET WebAPI маршрутизація визначається через атрибути або конвенцію.

Attribute routing: [Route("api/content")] на контролері, [HttpGet("{id}")] на методі.

Як працює: при старті додатку всі маршрути реєструються у таблиці маршрутів. При запиті URL порівнюється з шаблонами — знаходиться перший збіг, визначається контролер і метод, параметри витягуються з URL, query або тіла.

В Express (цей проект): аналогічно — router.get("/:id", handler) реєструє шаблон. :id — route parameter. Express проходить по зареєстрованих маршрутах зверху вниз і викликає перший збіг. Саме тому /search зареєстрований до /:id — інакше "search" розпарсилось би як id.
22. Як визначається, який контролер та який метод будуть виконуватися у відповідь на запит?
Визначення відбувається у два кроки:

1. Вибір контролера — за prefix URL:
— /api/content/* → ContentController (зареєстрований у server.ts через app.use("/api/content", createContentRouter(...)))
— /api/storages/* → StorageController

2. Вибір методу — за HTTP-метод + залишок URL:
— GET / → router.get("/") → getAll
— GET /search → router.get("/search") → search
— GET /1 → router.get("/:id") → getById
— POST / → router.post("/") → create
— PATCH /7/availability → router.patch("/:id/availability") → setAvailability

В ASP.NET це робить Action Selector на основі атрибутів [HttpGet], [HttpPost] та шаблону маршруту.
